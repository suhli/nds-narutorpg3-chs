# 字体绕过 VRAM 稳定性复核

更新时间：2026-05-27

## 修正点

此前把 `running=0 ARM9_PC=0x02089210 ARM7_PC=0x00000020` 记录为“替换 glyph 后停机”。复核后应修正为：这是调试暂停/断点现场，不能作为崩溃证据。

执行 `nds_resume` 后可继续运行：

```text
running=1 ARM9_PC=0x0200821C ARM7_PC=0x038042B0
```

## 已确认能力

- `0208914C` 可以保存当前字符码。
- `02089190` 前的 hook 可以按当前字符码替换 `R0`。
- `020087BC` 可以从 ARM9 RAM 中读取 glyph 数据。
- `0x82CD` 的原 VRAM glyph：

```text
0x06882280 = 11111111 11111111 11111111 12311231 ...
```

与 RAM 副本一致：

```text
0x02074180 = 11111111 11111111 11111111 12311231 ...
```

## 当前判定

动态 glyph 缓存路线可继续推进。

后续重点从“能否绕开 VRAM 源地址”切换到“如何管理中文 glyph 数据来源和缓存槽”：

- 中文 glyph 仓库优先放 NitroFS 或普通 RAM，不默认常驻 VRAM。
- VRAM/渲染缓冲只保留当前绘制需要的 glyph。
- 短期使用 ARM9 空洞做小表和少量 glyph 原型。
- 不能再把 hook 追加到 overlay_0000 尾部，因为已验证会踩 BSS。
