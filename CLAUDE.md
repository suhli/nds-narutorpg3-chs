# CLAUDE.md

## 项目背景

这是一个 NDS 游戏《火影忍者 RPG3》汉化工程。

- 原版 ROM 放在 `rom/origin.nds`。
- 逆向发现的文档放在 `hack/` 目录下。
- 汉化、逆向的工程计划放在 `plan/` 目录下；每次执行前需要先检查 `plan/`，执行完成后需要回写 `plan/`。
- 必要的新发现还需要同步写入 `hack/` 目录下。

## 工具和技能

### Python 环境

Python 环境已经使用 `uv venv` 初始化。

依赖安装规则：

- 需要把依赖记录为项目依赖时，使用 `uv add <package>`。
- 需要向当前虚拟环境安装依赖时，使用 `uv pip install <package>` 或 `uv pip install -r <requirements-file>`。
- 不要默认使用系统 `python` 或全局 `pip`；优先使用仓库内 `.venv` 和 `uv`。

### ndstool ROM 解包与打包

仓库内可以使用 `tools/ndstool.exe` 解包和打包 NDS ROM。若任务描述中写成 `tools/ndstools.exe`，以仓库内实际存在的 `tools/ndstool.exe` 为准。

相关流程已经做成项目 skill：`.claude/skills/ndstool-rom-workflow/SKILL.md`。需要查看、解包、打包或修复 ROM 头信息时，优先按这个 skill 的流程执行。

硬性约束：

- `rom/origin.nds` 是原版 ROM，只能作为输入读取，不能覆盖、原地 patch 或作为打包输出。
- 所有修改后的 ROM 都必须打包成新的 ROM 文件，例如 `rom/narutorpg3_chs.nds` 或任务专用测试文件名。
- 解包目录、回包命令、输出 ROM 名称和验证结果需要记录回 `plan/`。

## 汉化总体节奏

### 1. 先解决中文字库和字模承载方案

当前优先级最高的是字体系统，不要先进入大规模文本翻译。

已知字体加载入口和 VRAM 相关发现见 `hack/字体加载入口.md`：

- 字体加载核心函数：`FUN_overlay_0__02086870`
- NitroFS 文件读取函数：`FUN_overlay_0__0207f80c`
- 当前字体 tile 的 VRAM 映射地址：`0x06880000`
- 字体系统全局结构体：`DAT_020B73E0`

默认研究方向是：先尝试绕开“把完整字库和字模从 VRAM 加载/常驻”的方案。

原因是汉化需要的汉字数量远多于日文字库，完整中文字模很可能塞不进当前 VRAM 字体区域。不要把“扩充并一次性塞入 VRAM”作为默认方案，只能作为对照或临时验证。

优先尝试的方向：

- 追踪文本绘制函数如何通过 `tbl` 找到 tile index，并确认是否能改为新的字形索引流程。
- 研究是否可以让字库/字模主要放在 ROM 或普通 RAM 中，VRAM 只保留当前画面需要的字形缓存。
- 研究按需加载、分页缓存、动态替换 tile、常用字预载等方案。
- 确认 1x1、1x2 字体、调色板、tile index、字宽/排版逻辑之间的依赖关系。
- 每个可行方案都需要记录需要 patch 的地址、数据结构变化、容量上限和风险。

这个阶段的产物：

- 在 `hack/` 中补充字体绘制链路、字库/字模承载方案、关键函数和关键地址。
- 在 `plan/` 中记录选定方案、验证步骤、剩余风险。

### 2. 再 dump 出所有日文语句

字体方案明确后，再进入文本抽取阶段。

目标是 dump 出游戏内所有需要汉化的日文语句，并保留足够的回写定位信息。

需要关注：

- 从 ROM、NitroFS 文件、overlay、脚本、表格或压缩资源中定位文本。
- 区分剧情对白、菜单、系统提示、战斗文本、道具/技能/人物名等文本类别。
- 为每条文本保留来源文件、偏移、原始字节、编码、长度限制、控制符和上下文。
- 不要在不理解控制符和文本边界的情况下直接批量替换。

这个阶段的产物：

- 可复现的 dump 工具或脚本。
- 原文导出文件。
- 文本格式说明，包括编码、控制符、换行、占位符和长度限制。

### 3. 最后汉化并回写

完成字体方案和日文 dump 后，再进行翻译、校对、编码转换和 ROM 回写。

需要关注：

- 汉化文本必须保留控制符、变量占位符、换行和文本结束符。
- 回写前要确认每条文本的容量限制；超长文本需要走指针扩容、文本池迁移或脚本重排方案。
- 回写后需要重新打包 ROM，并在模拟器或真机环境中验证显示、换行、菜单布局和流程稳定性。
- 所有回写规则、工具参数和验证结果都要写回 `plan/`；新的逆向发现写入 `hack/`。

这个阶段的产物：

- 汉化文本表。
- 回写工具或脚本。
- 可运行的汉化 ROM。
- 显示测试和流程测试记录。
