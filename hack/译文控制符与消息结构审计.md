# 译文控制符与消息结构审计

## 结论

全量写回不能只比较原始 `jp_text` 与译文控制符。部分 `.m` 消息行的开头含脚本结构前缀，dump 会把这些字节误识别成 `{CTRL_xxxx}` 或 ASCII 字符。

当前审计规则以“调整后原文”为基准：

- `81 75` 开引号前的消息结构字节只保留在 ROM，不进入译文；
- `xx 00 + SJIS正文` 形态中，`xx 00` 是消息前缀，正文从后面的 SJIS 字节开始；
- 固定槽末尾的 `00` padding 不要求译文写成 `{CTRL_0000}`；
- 译文必须保留调整后正文中的真实 `{CTRL_xxxx}` 顺序；
- 当译文控制符列表与原始引号正文控制符列表完全一致时，审计器可以直接采用原始
  正文控制序列；不能用启发式结果删除注音/高亮控制对；
- 文本变短时仍由写回层在结束符前补 `81 40`，不提前移动 `03 00`。
- 无 `03 00` 结束符的 UI 固定槽不能一律用 `00` 补齐。`msg/item_msg.msg`、`msg/equip_msg.msg`、`msg/jyutu_msg.msg` 等说明表会把尾部 `00` 显示成异常符号，短译文应在写回层用 `81 40` 全角空格补齐；文本内部真实 `{CTRL_0000}` 分隔仍保持 `00 00`。
- 部分说明消息流以原始末尾 `00 00 00 00` 作为真实 `NUL4` 结束标记。该序列不能
  当成 padding 覆盖；短译文的全角空格必须填在它之前，并保持其原始末尾位置。
- 合并说明记录中的 `CTRL_0000` 是固定子槽位边界。道具、忍术、技能、队长效果及
  部分菜单说明必须按原始子槽位偏移原位写回，不能把整条译文压到记录开头。

## 写回联动

`tools/build_text_writeback_smoke_rom.py` 已同步支持两类消息前缀：

- `preserve_prefix_before_open_quote`
- `preserve_prefix_before_sjis_text`

这样报告中的调整可以直接用于 ROM 构建，不会在构建阶段再次覆盖消息前缀。

同一脚本已对已知 UI 说明表追加无结束符固定槽填充策略：

```text
fixed_slot_without_terminator=space_pad_known_ui_text_tables_else_zero_fill
```

写回器还支持两类说明结构保护：

```text
message_stream_nul4=preserve_original_end_marker_and_pad_before_it
fixed_message_subslots=write_each_translation_segment_at_original_offset
```
