# 菜单文本批量翻译与回写备忘

## 结论

标题菜单中文已显示，说明 overlay 菜单文本可以复用当前中文码表和 1x2 动态字体路径。之前方块字的关键原因不是字模文件完全错误，而是延迟缓存策略在标题菜单首次绘制时不能立即提供缺失字形。

当前可工作的测试 ROM 使用 `immediate_copy_on_1x2_miss_probe`。它验证了 1x2 首次 miss 同步填缓存的方向，但还不是完整正式字体补丁，1x1 路径仍需后续收敛。

## 菜单资产处理流程

1. 扫描 ARM9 和 overlay 中的 CP932 固定槽字符串：

```text
tools/scan_menu_overlay_strings.py

text/menu/overlay_menu_scan_report.json
```

2. 从扫描候选中只选择连续菜单/提示表区批量处理：

```text

overlay_0001 offset >= 0x1EC60
overlay_0002 offset >= 0x128B8
overlay_0003 offset >= 0x42A60
overlay_0004 offset >= 0x5270
```

3. 菜单翻译单独维护，不混进正文翻译表：

```text
tools/prepare_overlay_menu_translations.py
text/menu/overlay_menu_translations.tsv
text/menu/overlay_menu_translation_report.json
```

当前批量菜单表：

```text

```

## overlay_0002 控制分隔菜单项

`overlay_0002` 的店铺/道具提示区不全是普通 `00` 结尾 C 字符串。以下短项后面紧跟 `01 00` 分隔符，扫描器不会自动选中，回写时也不能覆盖分隔符：

```text
0x12C0C はい
0x12C34 どうぐ
0x12C3C うる
0x12C64 ぶき
0x12C6A ぼうぐ
0x12C72 うる
```

这些行已作为 `manual_0100_delimited` 加入菜单候选表。`slot_len` 只覆盖可见文本字节，写回后必须保留后续 `01 00`。

`overlay_0002:0x128B8` 的获取道具提示原文前面有 11 个全角空格，用作物品名占位。该行译文需要保留前导占位，再写入 `获得了！`，否则可能破坏“物品名 + 获取提示”的布局。

## overlay_0003 漏扫固定槽

`overlay_0003:0x43D64` 的 `だいじなもの` 位于 `みる`、`つかう` 同一段菜单表中，但前面夹有指针字节，自动扫描没有收进去。该位置已手工加入候选：

```text
0x43D64 だいじなもの -> 重要物
slot_len=16
notes=manual_null_delimited
```

该槽后面是 `00 00 00 00` 和 `%s` 格式串，回写时只能覆盖 16 字节槽内内容，不能继续向后扩写。

## 翻译原则

- 按去首尾空格后的 `jp_key` 去重翻译，同一短语复用同一译文。
- 菜单译文优先短、直观、适配固定槽容量。
- 保留 `%s`、全角数字、`／`、`：`、`？` 等格式符和 UI 符号语义。
- 普通菜单项不手工补尾部空格，写回时零填充固定槽；多列字符串才保留必要全角空格。
- 若译文引入当前码表没有的新字，必须追加码表并重建字体文件，不能只写 overlay。

## 组合属性字段回退

v6 手工加入的两条状态属性记录不是普通连续菜单字符串：

```text
menu_overlay_0000_030270  overlay/overlay_0000.bin  0x30270
menu_overlay_0003_0450A0  overlay/overlay_0003.bin  0x450A0
```

当前译文把多个属性名合并编码成一段连续中文，丢失了运行时依赖的字段边界。用户在 v14 中观察到状态页和升级属性全部显示为重复异常字形，确认这两条记录不能按单一固定槽整体回写。

v15 已将两条记录标为 `blocked_null_delimited`，不再参与菜单写回，对应范围恢复原版字节。后续应先定位每个属性名的实际分隔方式、索引表或指针表，再按独立字段翻译；不能继续整体覆盖。
