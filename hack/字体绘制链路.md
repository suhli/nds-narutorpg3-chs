# 字体绘制链路（阶段记录）

更新时间：2026-05-27
状态：进行中（静态链路已基本定位，动态验证未完成）

## 1. 目标与范围

- 本文只记录当前已确认的逆向结果与待验证点。
- 目标是为“绕开整套字库/字模常驻 VRAM”的改造提供准确入口。
- 当前阶段仅包含静态分析，不包含 ROM 回写。

## 3. 关键函数与地址

overlay 基址（overlay_0000）：

- `0x0207E320`

关键函数：

- `FUN_overlay_0__02086870`（overlay 偏移约 `0x8550`）
- `FUN_overlay_0__0207F80C`（文件加载路径中的被调函数）
- `FUN_overlay_0__020866BC`（4 字节表项的二分查找）
- `FUN_overlay_0__0208671C`（字符 -> glyph 索引/地址包装）
- `FUN_overlay_0__0208916C`（调用 `0208671C` 后执行字形复制/绘制路径）

关键常量/区域：

- `0x06880000`（字体路径相关常量中出现，指向 VRAM 区域）
- `DAT_020B73E0`（字体上下文关键结构）
- 常量字符串区包含：
  - `font/font.plt`
  - `font/font_1x1.tbl`
  - `font/font_1x1.chr`
  - `font/font_1x2.chr`
  - `font/font_1x2.tbl`

## 4. 当前链路结论（已确认）

`02086870` 负责字体初始化主流程：

1. 通过 `0207F80C` 加载 `.chr/.tbl/.plt`。
2. `.chr` 数据被拷贝到 VRAM（调用 `0x02006C00` 路径）。
3. `.tbl` 数据驻留 RAM，并把相关指针/计数写入 `DAT_020B73E0`。

`0208671C` + `020866BC` 负责查表：

1. 以字符码为 key，在 4 字节表项中二分查找。
2. 命中后返回 glyph 索引，再结合模式（1x1/1x2）计算最终读取位置。
3. 下游绘制函数（含 `0208916C` 路径）据此搬运字形数据。

## 5. 结构字段修正（重点）

相对 `DAT_020B73E0`：

- `+0x20` / `+0x28`：更像是 `.tbl` 的 RAM 指针。
- `+0x24` / `+0x2C`：更像是 `.tbl` 项计数（size >> 2），不是指针。

这点与早期“`+0x24/+0x2C` 是 tbl 指针”的判断冲突，当前以后者为准。

## 6. 对 VRAM 绕过方案的直接意义

可优先尝试在“查字 -> 取 glyph”层做改造，而不是只改初始化：

- 候选 Hook 点：`0208671C` 返回后到实际复制前的路径。
- 思路：按需解码/搬运当前字符字形，而不是整库常驻 VRAM。
- 这样可以保留现有 tbl 检索机制，缩小改造面。
