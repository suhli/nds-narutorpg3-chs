# 剧情消息流结构与结束符

## 背景

全量文本回写后，新建存档可以进入游戏，但开场第一句剧情文本会自动跳过，显示内容也不正确。更换 `0x8840`、`0x9040` 字码区间后症状仍然存在，说明问题不只是字码区间。

## 第一句定位

用户截图给出的第一句来自：

```text
source_file=msg/fld/evt/000.m
row_id=txt_2ef7aa25_000010_0000
offset=0x10
```

原始字节开头为：

```text
41 00 00 00 02 02 00 00 46 00 81 75 ...
```

其中 `81 75` 是日文开引号，之前的 `41 00 00 00 02 02 00 00 46 00` 是消息流结构前缀，不是可见文本。旧 dump 把这段前缀混进了 `jp_text`，导致回写时把结构字节也当成文本替换。

## 结束符位置

原版第一句的 `03 00` 结束符在文件偏移 `0x60`。旧的全量回写 ROM 把第一句结束符提前到了 `0x3A`，后面补了大量 `00`。

`.m` 剧情消息流在 `03 00` 后紧跟下一条脚本命令。提前结束并在原位置前补零，会让解释器从错误位置继续读命令，表现为文本自动跳过或剧情流程异常。

## 当前回写规则

对 `category=message` 且原始 slot 以 `03 00` 结束的行，当前工具采用结构安全回写：

- 保留第一处原始开引号 `81 75` 之前的结构前缀；
- 只替换可见文本和原行内控制符载荷；
- 结束符 `03 00` 保持在原始 slot 末尾位置；
- 文本变短时，在结束符前用 `81 40` 全角空格填充，不再提前写 `03 00` 后补零。

## 已构建验证 ROM

```text
rom/narutorpg3_chs_full_writeback_menu_structsafe_v1.nds
rom/narutorpg3_chs_full_writeback_menu_safe9040_structsafe_v1.nds
```

两者静态检查都恢复了第一句 `03 00` 在 `0x60` 的布局，并通过 `ndstool -i` Header CRC OK / Banner CRC OK。

## 后续观察点

如果 `structsafe_v1` 不再自动跳过，则消息流结构与结束符位置是主要根因。若仍有乱码但流程不跳过，再用 `safe9040_structsafe_v1` 区分是 `0xF0xx` 码段兼容问题还是字体映射问题。
