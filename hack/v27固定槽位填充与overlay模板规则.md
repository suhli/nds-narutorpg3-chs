# v27 固定槽位填充与 overlay 模板规则

## 结论

v27 新增确认：同一个 `00 00 00 00` 或无终止固定槽位，在不同资源表中的语义不同。不能把所有短译文都统一改成“提前终止 + 零填充”。

当前安全规则：

- 剧情 message 默认保留原始终止位置，终止符前用全角空格 padding。
- 菜单类短说明在确认是普通 `03 00` 或菜单专用 NUL4 说明时，才允许译文后提前终止。
- 道具、忍术、装备、技能、体调效果说明表的 NUL4 边界按固定可见槽位处理，译文后用全角空格补到原始 NUL4。
- `param/*.dat` 名称槽位无终止符，按固定可见槽位处理，必须用全角空格补齐。
- overlay 模板区必须保留运行时占位符和控制字节，只替换可见文字。

## overlay_0002 获取道具模板

位置：

```text
overlay/overlay_0002.bin:0x12C80..0x12DB0
```

该区不是普通 overlay 菜单行，而是获取道具/满背包等运行时模板。已确认的结构要点：

- `%s`：运行时插入道具名或对象名，必须保留。
- `%c`：运行时字符参数，占位符必须保留。
- `01 FF`：该模板区内的换行、换页或子段控制。它不是普通文本表里的 `{CTRL_0001}`，不能改写为 `01 00`。
- `03 FF`：该模板区内的消息结束控制。它不是普通 message 结束符 `03 00`，必须保留。
- `%sが` / `%sを`：日文助词在占位符后显示，中文场景下可把助词字节替换成全角空格，但不能删除 `%s` 或改变后续控制字节位置。

v27 覆盖的可见后缀：

```text
はいっていた！ -> 获得了！
てにいれた！ -> 获得了！
もちものが　いっぱいです！ -> 道具已满！
```

所有替换都按原字节宽度补齐，避免移动后续模板和控制参数。

## NUL4 的分场景规则

`00 00 00 00` 不能再作为全局“安全提前终止符”使用。

已确认可以提前 NUL4 结束的范围：

```text
msg/menu/battle_result_msg.msg
msg/menu/item_menu_msg.msg
```

已确认应保留原始 NUL4 边界并全角空格 padding 的范围：

```text
msg/item_msg.msg
msg/jyutu_msg.msg
msg/equip_msg.msg
msg/skill_msg.msg
msg/taityou_kouka.msg
```

原因：这些说明表会把零填充区域继续当成可见槽位读取，`00` 可能渲染成 `@`、乱码或造成说明截断。

## 固定槽位与参数表

`param/*.dat` 的道具名、忍术名、装备名等槽位没有普通文本终止符。这里的长度本身就是槽位边界。

规则：

- 译文短于原始槽位时，用全角空格补齐。
- 不写入 `00` 作为可见槽位 padding。
- 不移动后续数值、图标、属性或价格字段。

这类槽位会直接影响列表、商店、装备页和底部说明联动。

## message 固定槽位

部分 message 行没有普通 `03 00` 终止符，也没有已知场景尾控制。此时仍必须按 message 文本处理：

- 控制符外的半角 ASCII 要转全角或改成中文短语。
- padding 使用全角空格，不使用 `00`。
- 已确认的结构尾部单字节参数例外保留，例如前轮确认的 `N`、`4`。

这条规则用于避免电影说明、特殊教程和事件提示在半角 ASCII 或零填充处被误读为控制参数。

## CTRL_0000 子槽位

合并记录里的 `CTRL_0000` 常是固定子槽位边界，不只是普通空字符。

规则：

- 每个子槽位按原始起止偏移分别写回。
- 不能把整条译文压到记录开头。
- 子槽位内短译文按该槽位类型选择 padding。
- 分隔符和二进制尾部必须保持原始偏移。

## 后续排查提示

如果手测继续出现以下现象，优先按本文件规则分类：

- 显示 `@`：优先检查该槽位是否被 `00` padding。
- 半句后乱码：优先检查是否有半角 ASCII、占位符或控制字节被当作可见文本。
- 自动结束或颜色串到后续对话：优先检查 `03 00`、`03 FF`、高亮控制和场景尾控制是否被移动。
- 变量名前后残留日文助词：优先检查 `%s/%c` 或等价运行时占位符后缀。
- 大字标题或整块 UI 日文无法用 CP932 命中：先同时扫描原始文件和 LZ10 解压数据；若仍为 0 命中，再按 baked 图块、tile-index layout 或特殊 HUD 表排查。
