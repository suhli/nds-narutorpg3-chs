# 字体动态缓存方案

更新时间：2026-05-27
状态：第二版按字符码替换已验证，稳定性复核通过，进入动态缓存设计

## 1. 当前结论

原字体绘制链路已经动态确认：

```text
0208913C  文本绘制入口
0208671C  字符编码 + mode 查表，返回 glyph 源地址
02089190  调用复制函数，R0=源 glyph，R1=目标缓存，R2=大小
```

当前标题/菜单样本走 `mode=1`：

```text
chr1x2_base = 0x06881C00
unit_size   = 0x40
```

## 2. 最小验证目标

第一版不做完整中文字库缓存，只验证：

- glyph 复制函数能否接受非原 VRAM 字库地址作为 `R0`。
- `02089190` 前替换 `R0` 后，画面是否显示自定义 glyph。

如果成立，后续可把 `R0` 指向“按需准备的 glyph cache”，从而避开全量中文字模常驻 VRAM。

## 3. 第一版 Hook

Patch 点：

```text
02089190: bl 020087BC
```

改为：

```text
02089190: bl hook
```

hook 行为：

```text
if R0 == 0x06881C00:
    R0 = custom_glyph_data
b 020087BC
```

实际落点：

- `02089190` 改为 `BL 0x0207411C`。
- `0x0207411C` 位于 ARM9 主程序零区，不占 overlay BSS。
- `0x06881C00` 是字符 `0x8140` 在 `font_1x2.tbl` 中返回的源地址。
- `custom_glyph_data = 0x0207413C`。
- `R1/R2` 保持原样。

已验证：

```text
hook 入口:
R0=0x06881C00
R1=0x02292A40
R2=0x00000040
LR=0x02089194

进入原复制函数 020087BC:
R0=0x0207413C
R1=0x02292A40
R2=0x00000040
LR=0x02089194
```

结论：复制函数接受非原 VRAM 字库地址作为 glyph 源。这条路可以继续扩展成“按需准备 glyph 数据，再替换 `R0`”的动态缓存方案。

新增两个 hook：

```text

```

`0207411C` 保存当前字符码：

```text
mov r6, r1
*(u32*)0x020741C0 = r1
```

`02074140` 在复制前读取当前字符码：

```text
if (*(u32*)0x020741C0 == 0x82CD):

```

MCP 已验证：

```text
current_char=0x82CD
R0=0x02074180
PC=0x020087BC
R1=0x02292B40
R2=0x00000040
```

结论：

- 当前字符码可以从 `0208914C` 保存。
- `02089190` copy hook 可以按字符码选择 glyph 源。
- 这已经接近正式动态缓存所需的控制面：`char_code -> glyph_source`。

风险：

- 此前记录的 `running=0 ARM9_PC=0x0208AA44 / ARM7_PC=0x00000020` 可通过 `nds_resume` 继续运行，不能作为崩溃证据。
- no-op 复核已经完成：保存字符 hook 稳定，copy hook 框架稳定。
- “替换为原 glyph 拷贝”版本已经复核：`0x02074180` 与 `0x06882280` 数据一致，清除断点后可继续运行。
- 下一步进入动态 glyph 缓存设计，重点转为中文 glyph 数据来源、映射表和缓存槽管理。

## 4. 后续正式方向

第二版稳定后，再扩展为：

- 在 `0208671C` 或 `02089190` 处识别中文字符。
- 从 ROM/RAM 字模仓库按需取 glyph。
- 将 glyph 放入小型缓存区。
- 返回/替换为缓存区地址。
- 维护字符到缓存槽的映射和淘汰策略。

需要同时考虑：

- `.chr` 容量。
- `.tbl` 容量。
- 1x1/1x2 mode。
- 同屏复用和文本刷新。

## 10. RAM 预加载容量判断

新增压力测试脚本：

```text
tools/run_font_preload_size_sweep.py
```

验证对象仍是 `font/chs_probe.bin`，但把文件总大小扩大到不同档位。文件头和两个测试 glyph 保持一致，后续只做填充，用于观察整文件预加载对 RAM 的压力。

关键结果：

```text
512K/896K/960K/992K/1008K 可进入文本绘制采样。
1008K: 82CD -> R0=0x02286B80, 82DF -> R0=0x02286BC0。
1024K: 分配后推进到文本流程失稳，未取得正常绘制样本。
1392K+: chs_data_ptr=0，连续 RAM 分配失败。
```

判断：

- `0207F80C` 不会自动把大文件分页加载；当前用法是整文件连续加载到 RAM。
- 文件太大时会出现 `chs_data_size` 被写入但 `chs_data_ptr=0` 的失败形态。
- 即使拿到 pointer，接近 1MB 时也可能压缩游戏后续运行空间。
- 正式方案不能把完整中文字模设计成单个大文件常驻 RAM。

映射表容量判断：

```text
8 bytes/entry 时：
10000 字约 80KB
20000 字约 160KB
```

映射表本身不太可能先超过 RAM；优先需要控制的是 glyph 数据。1x2 glyph 当前是 `0x40 bytes/glyph`，10000 个 glyph 约 640KB。后续应采用：

```text
常驻 map/header
按页或按块加载 glyph 数据
小型 RAM staging buffer
当前画面 VRAM glyph cache
```

## 11. 1x1 字体路径验证

新增测试 ROM：

```text
rom/test_vram_font_file_preload_1x1_probe.nds
```

构建参数：

```text
tools/patch_vram_font_file_preload_probe.py
--char-a 0x82A2
--char-b 0x82BD
```

no-op 基线采样：

```text
current_char=0x8140
R0=0x06880000
R1=0x06894000
R2=0x00000020
LR=0x02089194
```

这确认 1x1 也从 `02089190` 调用 `020087BC`，只是复制大小为 `0x20`，源基址为 `font_1x1.chr = 0x06880000`。

RAM glyph 替换采样：

```text
current_char=0x82BD R0=0x02282FA0 R2=0x20
current_char=0x82A2 R0=0x02282F60 R2=0x20
```

结论：

- `02089190` copy hook 可以同时服务 1x1 和 1x2。
- 1x1 glyph 大小为 `0x20`，1x2 glyph 大小为 `0x40`。
- 1x1 glyph 源可以替换到 NitroFS 预加载后的普通 RAM 数据。

正式格式约束：

当前 probe 只按 `char_code` 查表，导致 `0x82A2` 在 1x2 样本中也会被替换到同一个 RAM glyph。正式实现不能只用字符码做 key，需要：

```text
char_code + mode
```

或者拆成：

```text
chs_1x1.map / chs_1x1.chunk
chs_1x2.map / chs_1x2.chunk
```

考虑到 ARM9 hook 空间有限，优先建议使用两套 map/chunk，运行时根据 `R2` 或已知 mode 选择查找表。


map header：

```text
0x00 char[4] magic
0x04 u16 version
0x06 u16 header_size

0x0C u32 entry_count
0x10 u32 flags
0x14 u32 default_chunk_id
0x18 u32 reserved0

```

map entry：

```text
0x00 u32 char_code
0x04 u32 glyph_offset
0x08 u16 advance
0x0A u16 flags
0x0C u16 chunk_id
0x0E u16 reserved
```

chunk header：

```text
0x00 char[4] magic
0x04 u16 version
0x06 u16 header_size
0x08 u16 glyph_size
0x0A u16 compression
0x0C u32 glyph_count
0x10 u32 data_size
0x14 u32 flags
0x18 u32 reserved0
0x1C u32 reserved1
```

当前 hook 行为：

```text
entry_count = *(map + 0x0C)
entry = map + 0x20
if entry.char_code == current_char:
    R0 = chunk + entry.glyph_offset
```

运行时验证：

```text
0x82A2, R2=0x40 -> R0=0x02283100
0x82A2, R2=0x20 -> R0=0x02283000
0x82CD, R2=0x40 -> R0=0x02283140
0x82DF, R2=0x40 -> R0=0x02283180
0x82BD, R2=0x20 -> R0=0x02283020
```

结论：

- `CHMP/CHCK` formal v0 可作为后续动态字体文件格式基础。
- `glyph_offset` 当前从 chunk 文件起点计算，第一枚 glyph offset 为 `0x20`。
- `advance/flags/chunk_id` 已预留，但当前 hook 暂不消费；下一步围绕 chunk 分页、缺字 fallback 和查找性能继续设计。

## 14. chunk_id fallback 原型

新增测试 ROM：

```text
rom/test_vram_font_chunk_fallback_probe.nds
```

新增脚本：

```text
tools/patch_vram_font_chunk_fallback_probe.py
```

该版本保持 formal v0 map/chunk 结构不变，但开始使用 entry 内的 `chunk_id` 字段：

```text
chunk_id == 0 -> R0 = chunk_base + glyph_offset
chunk_id != 0 -> R0 = chunk_base + 0x20
```

`0x20` 是 chunk header 大小，当前被约定为每个 chunk 的 fallback glyph offset。

MCP 采样：

```text
0x82CD, R2=0x40 -> R0=0x02283120  fallback
0x82BD, R2=0x20 -> R0=0x02283000  fallback
0x82A2, R2=0x40 -> R0=0x02283160  resident
0x82A2, R2=0x20 -> R0=0x02283020  resident
0x82DF, R2=0x40 -> R0=0x022831A0  resident
```

结论：
- `chunk_id` 字段已经可以在 `02089190` copy hook 中参与判断。
- 未驻留 chunk 可以先回退到显式 fallback glyph，而不是继续使用原日文字形。
- 正式实现应把 `chunk_id != 0` 的路径扩展为查询 chunk table、按需加载目标 chunk，失败时再 fallback。

```text
0x020743D4  resident_1x1_chunk_id
0x020743D8  resident_1x2_chunk_id
```

MCP 样本：

```text
0x82DF -> R0=0x02283120, miss_flag=1, miss_chunk_id=0, resident_1x2=1
next entry -> miss_flag=0, resident_1x2=0
0x82A2 -> R0=0x02283160, resident hit
```

结论：
- `0208913C` 可作为 copy hook 外的 miss 消费点。
- 当前只验证 resident slot id 更新，尚未真实换入 chunk 数据。
- 后续 loader 应在该层或更高层消费 miss，而不是放在 `02089190` copy hook 中。

## 20. resident copy v2 原型

新增 probe：

```text
tools/patch_vram_font_chunk_table_resident_copy_probe.py
rom/test_vram_font_chunk_table_resident_copy_v2_probe.nds
```

v1 曾把 resident buffer 放在固定 ARM9 空洞，但运行时样本显示该区域会被其他数据污染。v2 改为把 resident page 放在 `chs_1x2.chunk` 自己的 heap buffer 中：

```text
0x000  CHPK header
0x020  resident page
0x100  source page 0
0x1E0  source page 1
```

consumer 仍挂在：

```text
0x0208913C -> 0x02073D64
consume_hook_size = 0x90
```

MCP 样本：

```text
1x2_chunk_ptr = 0x02283100
resident page = 0x02283120
0x82DF -> R0=0x02283140, data=C77CC77C C77CC77C, miss=1/82DF/0/0x40
next entry -> miss_flag=0, resident_1x2=0
0x82A2 -> R0=0x02283180, data=73377337 73377337
```

结论：
- `0x0208913C` 层可以完成真实 page 搬运，后续 copy hook 会读到换页后的 resident glyph。
- resident buffer 应归属 chunk heap，不应依赖固定 ARM9 空洞。
- 单 1x2 resident slot 会在 chunk 0/1 之间抖动；下一版需要验证双 slot 或文本块预扫。

## 21. dual-slot 1x2 原型

新增 probe：

```text
tools/patch_vram_font_chunk_table_dual_slot_probe.py
rom/test_vram_font_chunk_table_dual_slot_v2_probe.nds
```

该版本把 1x2 resident 扩为两个 slot：

```text
0x020  resident slot0, initial chunk1
0x100  resident slot1, initially invalid
0x1E0  source page0
0x2C0  source page1
```

copy hook 主体迁到远端空洞：

```text
0x02074140  copy trampoline
0x02073D64  copy hook body, size=0xCC
0x020743E4  consume hook, size=0xB8
```

MCP 样本：

```text
0x82CD -> R0=0x022831C0, data=95599559 95599559, slot0=1, slot1=FFFFFFFF
0x82DF -> R0=0x02283140, data=C77CC77C C77CC77C, miss=1/82DF/0/0x40
0x82A2 -> R0=0x02283260, data=73377337 73377337, slot0=1, slot1=0
0x82C6 -> R0=0x022831C0, data=95599559 95599559, slot0=1, slot1=0
```

结论：
- 双 slot 可以让 chunk0 装入 slot1，同时保留 slot0 的 chunk1。
- 这消除了单 slot 在 chunk0/chunk1 间来回 miss 的问题。
- 当前只验证 1x2；正式实现还要补 1x1、多 chunk 替换策略和更稳妥的代码区规划。

## 22. dual-mode 1x1/1x2 原型

新增 probe：
```text
tools/patch_vram_font_chunk_table_dual_mode_probe.py
rom/test_vram_font_chunk_table_dual_mode_probe.nds
```

该版本的 copy hook 同时处理两种字体：

```text
R2=0x20 -> chs_1x1.map / chs_1x1.chunk
R2=0x40 -> chs_1x2.map / chs_1x2.chunk
```

关键样本：
```text
0x8140, R2=0x40 -> R0=0x02284B60, data=84488448 84488448
0x8140, R2=0x20 -> R0=0x02283040, data=41144114 41144114
0x82BD, R2=0x20 -> R0=0x02283060, data=62266226 62266226
0x82DF, R2=0x40 -> miss=1/82DF/0/0x40
0x82A2, R2=0x40 -> R0=0x02284C40, data=73377337 73377337
0x82C6, R2=0x40 -> R0=0x02284BA0, data=95599559 95599559
```

结论：
- 同一字符码 `0x8140` 在 1x1/1x2 下命中不同图样，说明 split-map + chunk table 已解决两种字体互相 fallback/missing 的核心问题。
- 1x2 dual-slot 策略在 dual-mode 版本中仍有效，没有被 1x1 分支破坏。
- 当前 1x1 仍按 `resident_1x1_chunk_id=0` 常驻；若正式中文字库需要更多 1x1 chunk，应补 `miss_mode=0x20` 的 consumer。

## 23. dual-mode 1x1 page copy 原型

新增 probe：
```text
tools/patch_vram_font_chunk_table_dual_mode_1x1_copy_probe.py
rom/test_vram_font_chunk_table_dual_mode_1x1_copy_probe.nds
```

consumer 代码区改为：
```text
0x020743E4  trampoline
0x020718D8  body, size=0xFC
```

关键样本：
```text

0x82A2, R2=0x40 -> R0=0x02284C40, data=73377337 73377337
0x82C6, R2=0x40 -> R0=0x02284BA0, data=95599559 95599559
```

结论：
- 1x1 的 `miss_mode=0x20` 已可触发真实 resident page copy。
- 1x1/1x2 两种字体现在都验证了独立 map、独立 resident 状态、独立 miss consumer。
- 当前 1x1 copy 分支已支持 `chunk_id=0/1` 两页选择；正式实现若要支持更多 chunk，需要通用 source page 表或更大的 consumer 代码区。

## 29. v19 双模式延迟换页失败与 v20 同步双模式

用户运行 `rom/narutorpg3_chs_patcher_v19_allfix_rebuild_fusion12.nds` 后确认，标题菜单等 1x2 文本全部显示为占位方框。截图记录：

```text
plan/cache/text-writeback-smoke/v5-regression-20260603/v19-issue-1x2-font-missing.jpg
```

v19 虽然同时识别 `R2=0x20` 和 `R2=0x40`，但两种模式在 cache miss 时都先返回占位字形，再依赖绘制入口 consumer 换页。标题菜单的 1x2 文本没有发生有效重绘，因此只保留第一次返回的占位方框。

兼容方案改为同步双模式：

```text
patch_vram_font_chunk_table_dual_immediate_cache_probe.py
```

执行路径：

```text
0x02074140:
  R2=0x20 -> 0x020718D8  1x1 同步换页
  其他    -> 0x02073D64  旧版 1x2 同步双槽换页
```

关键约束：

- 1x2 hook 完整复用旧即时缓存实现，不改变其指令；
- 1x1 使用独立同步换页助手，不依赖绘制入口重绘；
- overlay 绘制入口 `0x0208913C` 保持原版指令；
- 1x1/1x2 均在当前取字调用内返回真实字形地址。

`rom/narutorpg3_chs_patcher_v20_dual_immediate_fusion12.nds` 静态验证：

- 实际 1x2 hook 与 v15 已运行显示正常的 1x2 hook 逐字节相同；
- 分流入口 12 字节，1x1 同步助手 160 字节，1x2 hook 268 字节，三段互不重叠；
- 实际 ROM 中三段 hook 均与生成结果逐字节一致；
- overlay 绘制入口与原版一致，不再安装延迟 consumer；
- 1x1/1x2 字体数据与 v19 一致，1x2 回归仅由执行路径引起。

运行时仍需确认 v20 的标题菜单 1x2 文本和状态/升级/商店/保存选项 1x1 文本能同时显示。
