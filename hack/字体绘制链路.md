# 字体绘制链路（阶段记录）

更新时间：2026-05-27
状态：进行中（字体加载、查表和 glyph 复制路径已动态闭合）

- 目标是为“绕开整套字库/字模常驻 VRAM”的改造提供准确入口。

## 3. 关键函数与地址

overlay 基址（overlay_0000）：

- `0x0207E320`

关键函数：

- `FUN_overlay_0__02086870`（overlay 偏移约 `0x8550`）

关键常量/区域：

- 常量字符串区包含：
  - `font/font.plt`

  - `font/font_1x1.chr`
  - `font/font_1x2.chr`
  - `font/font_1x2.tbl`

## 4. 当前链路结论（已确认）

`02086870` 负责字体初始化主流程：

1. 通过 `0207F80C` 加载 `.chr/.tbl/.plt`。
2. `.chr` 与 `.tbl` 顺序加载到 `0x06880000` 开头的 VRAM 字体区域。
3. `font.plt` 加载到普通 RAM/堆地址。
4. 初始化后把 VRAM 指针、RAM 指针和计数写入 `DAT_020B73E0`。

`0208671C` + `020866BC` 负责查表：

1. 以字符码为 key，在 4 字节表项中二分查找。
2. 命中后返回 glyph 索引，再结合模式（1x1/1x2）计算最终读取位置。
3. 下游绘制函数（含 `0208916C` 路径）据此搬运字形数据。

动态已确认 `0208913C -> 0208916C -> 0208671C -> 02089170 -> 02089190` 是实际绘制链路：

- `0208913C` 入口 `R1` 为当前字符编码。
- `0208671C` 输入 `R0=字符编码`，`R1=mode`。
- `0208671C` 返回 `R0=glyph_source`。
- `02089190` 调用复制函数时 `R0=glyph_source`、`R1=目标缓存`、`R2=复制大小`。
- 标题/菜单样本走 `mode=1`，复制大小为 `0x40`。

## 5. 结构字段修正（重点）

相对 `DAT_020B73E0`，运行时已观察到：

- `+0x00 = 0x06880000`：`font_1x1.chr` VRAM 基址。
- `+0x04 = 0x06881C00`：`font_1x2.chr` VRAM 基址。
- `+0x08 = 0x02282EA0`：`font.plt` RAM/堆指针。
- `+0x0C = 0x00000004`：palette 相关计数。
- `+0x20 = 0x0688E6C0`：`font_1x1.tbl` VRAM 指针。
- `+0x24 = 0x000000E0`：`font_1x1.tbl` 项计数（896 / 4）。
- `+0x28 = 0x0688EA40`：`font_1x2.tbl` VRAM 指针。
- `+0x2C = 0x0000032B`：`font_1x2.tbl` 项计数（3244 / 4）。

这点修正了早期两个判断：

- `+0x24/+0x2C` 不是 tbl 指针，而是 tbl 项计数。
- `+0x20/+0x28` 是 tbl 指针，但指向 VRAM，不是普通 RAM。

运行时读取 `0x0688E6C0` 与 `0x0688EA40`，其前 64 字节分别与 `font_1x1.tbl`、`font_1x2.tbl` 文件开头一致。

## 6. 对 VRAM 绕过方案的直接意义

可优先尝试在“查字 -> 取 glyph”层做改造，而不是只改初始化：

- 候选 Hook 点 1：`0208671C`，负责从字符码查表并返回 glyph VRAM 地址。
- 候选 Hook 点 2：`02089190` 前后，负责把 glyph 源地址复制到目标缓存。
- 思路：保留原查表模式，优先让返回地址指向一个动态 VRAM glyph cache。
- 需要注意：原 `.tbl` 当前也在 VRAM 中。如果中文 `.tbl` 明显变大，也要一起避开“全量 tbl 常驻 VRAM”的问题，不能只处理 `.chr`。
