# 译文控制符与消息结构审计

## 结论

全量写回不能只比较原始 `jp_text` 与译文控制符。部分 `.m` 消息行的开头含脚本结构前缀，dump 会把这些字节误识别成 `{CTRL_xxxx}` 或 ASCII 字符。

当前审计规则以“调整后原文”为基准：

- `81 75` 开引号前的消息结构字节只保留在 ROM，不进入译文；
- `xx 00 + SJIS正文` 形态中，`xx 00` 是消息前缀，正文从后面的 SJIS 字节开始；
- 固定槽末尾的 `00` padding 不要求译文写成 `{CTRL_0000}`；
- 译文必须保留调整后正文中的真实 `{CTRL_xxxx}` 顺序；
- 当译文控制符列表与原始引号正文控制符列表完全一致时，审计器可以直接采用原始
  正文控制序列；不能用启发式结果删除注音/高亮控制对；
- 文本变短时仍由写回层在结束符前补 `81 40`，不提前移动 `03 00`。
- 无 `03 00` 结束符的 UI 固定槽不能一律用 `00` 补齐。`msg/item_msg.msg`、`msg/equip_msg.msg`、`msg/jyutu_msg.msg` 等说明表会把尾部 `00` 显示成异常符号，短译文应在写回层用 `81 40` 全角空格补齐；文本内部真实 `{CTRL_0000}` 分隔仍保持 `00 00`。
- 部分说明消息流以原始末尾 `00 00 00 00` 作为真实 `NUL4` 结束标记。该序列不能
  当成 padding 覆盖；短译文的全角空格必须填在它之前，并保持其原始末尾位置。
- 合并说明记录中的 `CTRL_0000` 是固定子槽位边界。道具、忍术、技能、队长效果及
  部分菜单说明必须按原始子槽位偏移原位写回，不能把整条译文压到记录开头。

## 写回联动

`tools/build_text_writeback_smoke_rom.py` 已同步支持两类消息前缀：

- `preserve_prefix_before_open_quote`
- `preserve_prefix_before_sjis_text`

这样报告中的调整可以直接用于 ROM 构建，不会在构建阶段再次覆盖消息前缀。

同一脚本已对已知 UI 说明表追加无结束符固定槽填充策略：

```text
fixed_slot_without_terminator=space_pad_known_ui_text_tables_else_zero_fill
```

写回器还支持两类说明结构保护：

```text
message_stream_nul4=preserve_original_end_marker_and_pad_before_it
fixed_message_subslots=write_each_translation_segment_at_original_offset
```

## 2026-06-05 v23 规则修订

用户手动测试 v22 后确认，说明类 NUL4 记录如果把短译文后面的剩余容量全部填成 `81 40`，运行时会把这些空格当作可显示正文，表现为空白首屏、需要再次按 A/B 才看到说明，或在道具/忍术说明尾部出现占位符。

因此 v23 将规则改为：

```text
known_description_nul4=write_translation_then_00_00_00_00_then_zero_fill
```

当前适用来源：

```text
msg/equip_msg.msg
msg/item_msg.msg
msg/jyutu_msg.msg
msg/menu/item_menu_msg.msg
msg/skill_msg.msg
msg/taityou_kouka.msg
```

`03 00` 结尾的剧情 message 仍保留原结束符位置，不提前移动 `03 00`。对少数控制位敏感对白，使用更窄的规则：

```text
fixed_control_slots=write_each_segment_at_original_control_offset
```

当前启用行：

```text
zh_txt_2ef7aa25_000758_0027
zh_txt_6b929156_0001A6_0005
zh_txt_6b929156_0001FA_0006
```

合并说明记录继续按 `CTRL_0000` 原始子槽位原位写回。v23 已扩展到成员、保存、商店、技能、阵型、通灵、Wi-Fi 好友等菜单说明记录；静态验证中 36 条命中该策略且 separator 偏移未移动。

## 2026-06-05 v24 规则修订

### overlay 固定选项

保存确认一类 overlay_0002 固定选项行不能只把译文写入后立刻 `00` 填充。运行时会把第一段后的 `00 00` 视为选项列表终止，导致后续“否”不显示。

当前写回规则：

```text
overlay_fixed_visible_span=pad_visible_span_to_original_raw_len_with_8140_then_zero_fill_slot_tail
```

目标是保留原本选项分隔、颜色和后续参数的相对位置，同时避免可见文本段内部出现提前终止。

### 固定子槽位 padding

`CTRL_0000` 合并记录中的每个子槽位仍必须按原始绝对偏移写回，但译文后的剩余容量不能再统一填 `81 40`。菜单说明、好友说明和部分道具/忍术说明会把这些空格继续当成正文渲染，表现为空白首屏、需要按 A/B 后才显示，或显示 `@` 占位符。

当前写回规则：

```text
fixed_message_subslots=write_each_segment_at_original_offset_then_zero_fill_inside_subslot
```

### 菜单说明结束符

仅对菜单/好友说明类普通 `03 00` 记录启用提前结束：

```text
early_03_terminator_sources=msg/menu/*, msg/wifi/friend_msg.msg
```

写回时把 `03 00` 放在译文后，槽位尾部 `00` 填充。剧情 message 不使用该规则，仍保持原始 `03 00` 位置和已验证的安全 padding。

### 固定宽度 overlay 文本

装备页和状态页的 overlay 文本必须保持原始显示宽度。装备分类 `menu_overlay_0003_04430C` 当前固定写为：

```text
ぶき -> 武器
ぼうぐ -> 防具　
きゃはん -> 护腿　　
```

属性效果说明不属于 overlay，而来自 `msg/taityou_kouka.msg`。该表应使用“攻升/防降/防升/攻降/速升/速降”等短译，避免固定布局错位。
